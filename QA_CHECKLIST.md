### Scoring logic — quoted from `src/app/api/attempts/[attemptId]/submit/route.ts:24-126` (`scoreAnswers`)
- **MCQ / true_false** (lines 40-46): `const selectedOpt = q.options?.find(o => o.id === answer); correct = selectedOpt?.isCorrect === true;` — ID-based lookup, all-or-nothing.
- **fill_blank / short_answer** (47-54): `answer.trim().toLowerCase() === q.correctAnswer.trim().toLowerCase()` — case/whitespace-insensitive exact match, all-or-nothing.
- **MRQ** (55-65): sorted selected option IDs vs sorted correct option IDs, exact set match required (no partial credit for MRQ — all-or-nothing by design).
- **Matching** (66-95): **two code paths** —
  - New format (object `{leftOptionId: rightText}`): per-pair partial credit — `marksAwarded = (q.marks / q.options.length) * correctPairs`, rounded to 2 decimals via `parseFloat(...toFixed(2))`.
  - Legacy format (array of selected option IDs): all-or-nothing.
- **Ordering** (96-111): partial credit per correctly-positioned item — `marksAwarded = (q.marks / expected.length) * correctPositions`, same `toFixed(2)` rounding.
- **essay / coding / file_upload** (112-118): always `correct=false, marksAwarded=0` — manual/async grading only, matches "no /api/grade route" finding above.

## 3. FUNCTIONAL — Student

**STU-01** — [OBSERVED BUG, partially confirmed] Auto-start works for exam *availability*, but nothing enforces auto-*end*, and the raw `Exam.status` never transitions itself
Root cause, traced through `src/lib/data/analytics.ts:255-315` (`getStudentExams`):
```ts
let status: 'available' | 'upcoming' | 'completed' = 'upcoming';
if (submitted || exam.status === 'completed') {
  status = 'completed';
} else if (exam.status === 'live' || exam.startTime <= now) {
  status = 'available';
}
```
This correctly makes an exam "available" to the student purely from wall-clock time (`exam.startTime <= now`), regardless of whether a teacher manually flipped the DB `status` column — so **auto-start for the student-facing gate already works** without manual intervention. Confirmed no cron job or scheduled function exists anywhere in the repo (`vercel.json` has no `crons` key; grepped for any time-driven status-transition code in `src/lib/data` — none found).

What's actually broken:
1. **No upper bound.** This function never checks `exam.endTime`. An unsubmitted exam stays `'available'` forever past its scheduled end unless a teacher manually marks it `completed` — a student could join and start an attempt hours after the window closed.
2. **Raw `Exam.status` DB column never auto-transitions** `scheduled → live → completed` on its own. It only changes via explicit `PUT`/`PATCH` calls (teacher "Go Live" button, admin approval, etc.). This means teacher/admin dashboards showing the raw status badge go stale — an exam can be actively `available` to students (per the wall-clock derivation above) while the teacher's own exam list still shows "Scheduled."
3. **Per-attempt end-time enforcement does work** once a student is inside the exam page: `src/app/exam/[examId]/page.tsx:124-127` computes `remaining = Math.max(0, endMs - serverNow)`; if a student joins after `endTime`, `initialSeconds` is `0` and the timer (`useExamTimer`) should immediately trigger auto-submit — confirm this doesn't leave a 0-second stuck state.
Steps: Create exam with `endTime` = 2 minutes from now, approve it, wait past end time, then as a *new* student attempt to start the exam.
Expected: Exam should no longer be joinable / should show as ended.
Actual (predicted from code): Exam still shows "Available Now" indefinitely (bug #1 above) and a new attempt can technically be created via `POST /api/attempts` even after `endTime` — that route (`src/app/api/attempts/route.ts`) has **no endTime check at all**, only checks `role === 'student'`.
Priority: **P0** — data-corrupting (a late attempt gets `initialSeconds=0`, immediately auto-submits with zero answers, but still consumes the student's one attempt via the `[examId,studentId]` unique constraint with no way to retry). [AUTO] (scriptable via time manipulation / short-window exam + curl)

**SCR-05** — [NEW FINDING] Float marksAwarded into an `Int` column — possible submit-time crash
Root cause: `Answer.marksAwarded Int?` and `ExamAttempt.score Int?` in `prisma/schema.prisma:295,320`, but the matching/ordering partial-credit formulas (`submit/route.ts:83`, `:107`) compute `parseFloat(((q.marks / q.options.length) * correctPairs).toFixed(2))` — e.g. 8 marks over 3 pairs with 2 correct = `5.33`, a float. This value flows into `score += marksAwarded` (also becomes fractional) and is written via `prisma.answer.upsert({ data: { marksAwarded: a.marksAwarded } })` and `prisma.examAttempt.update({ data: { score } })`.
Steps: Create a matching question worth 8 marks with 3 pairs (not evenly divisible); submit with exactly 2/3 correct; observe the `POST .../submit` response.
Expected: Either the schema should be `Float?` to accommodate partial credit, or the scoring code should round to whole-number marks before persisting.
Actual (predicted): Prisma Client validates arg types against the schema before querying — passing a float where `Int` is expected typically throws a client-side `PrismaClientValidationError` inside the `$transaction`, meaning **the entire submit transaction fails and the student's exam never gets marked as submitted** (worse than a wrong score — the student is stuck, and `attempt.status` stays `in_progress` so they can't resubmit through the normal flow since `POST /api/attempts` upserts to the existing in-progress record).
Priority: **P0 — highest priority in this whole document.** Untested edge case that plausibly makes real matching/ordering questions with uneven mark distribution un-submittable. [AUTO] — trivial to script: create exam with an 8-mark/3-option matching question, submit 2/3 correct, assert the request succeeds.

- After end: confirmed no check (see STU-01).

**SEC-08** — RLS coverage — confirmed **zero** tables have database-level RLS
No `supabase/` directory, no SQL migrations, no `CREATE POLICY` statements anywhere in the repo. This is the underlying reason IDOR bugs are possible at all: there's no defense-in-depth layer.
Priority: P1 (architectural — not fixable as a quick patch, but worth deciding whether to invest in RLS policies for the highest-value tables: `Question`, `ExamAttempt`, `Answer`, `Exam`). [MANUAL] (needs a design decision)

**TIME-01** — [OBSERVED BUG] Timezone consistency across roles — **appears fixed on code inspection, needs live cross-timezone verification**
Traced the full pipeline: exam creation (`teacher/exams/new/page.tsx:286,292`) does `new Date(\`${d}T${time}\`).toISOString()` — parses the picked date+time as the *browser's local* timezone, converts to a UTC ISO string for storage. All render sites checked (`admin/page.tsx:335`, `admin/exams/page.tsx:214,216`, `teacher/exams/page.tsx:316`) use `new Date(storedUTCString).toLocaleString()`/`toLocaleDateString()`, which correctly re-renders in **each viewer's own** local timezone. `exam.date` (used in `admin/exams/page.tsx`) is confirmed to be the same underlying value (`date: e.startTime.toISOString()` in `analytics.ts:384,606`), not a separately-computed, possibly-stale field.
Important nuance to test for, not a bug: if admin and teacher are in genuinely different system timezones, they are *expected* to see different wall-clock strings for the same instant — that's correct, not the bug. The original bug (pre-fix) was storing/comparing un-normalized local-time strings inconsistently.
Steps: Set two browser sessions to different OS timezones (e.g. UTC and UTC+5); create/view the same exam in both; confirm both show the *correctly locally-translated* time for the same underlying instant (not two different instants).
